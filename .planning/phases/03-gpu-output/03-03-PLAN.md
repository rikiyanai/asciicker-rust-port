  <action>
1. **Add resize system** in `engine-port/src/output/mod.rs`:
   - Create the resize handler with the canonical Bevy 0.18 signature:
   ```rust
   fn handle_window_resize(
       resize_events: Option<MessageReader<WindowResized>>,  // Option for test-env compatibility
       mut grid: ResMut<AsciiCellGrid>,
       config: Res<AsciiRenderConfig>,
       windows: Query<&Window>,                               // physical pixels for HiDPI
   )
   ```
   *(Correction history: P3-007 renamed EventReader->MessageReader; P3-H03 added Option wrapper for test-env; P3-M04 added windows Query for physical pixels; R6-H02 consolidated.)*
   - For each resize event: compute `new_w = event.width as u32 / config.font_width` and `new_h = event.height as u32 / config.font_height`.
   - **P3-M04 FIX:** Do NOT use `event.width` / `event.height` for dimension calculation — they report logical pixels, which gives wrong results on HiDPI/Retina displays. Instead, accept `windows: Query<&Window>` as a system parameter and read `window.physical_width()` / `window.physical_height()`. Compute `new_w = physical_width / config.font_width`, `new_h = physical_height / config.font_height`. The WGSL shader's `@builtin(position)` operates in physical pixel space.
   - Guard against zero dimensions: if new_w == 0 or new_h == 0, skip.
   - If new_w != grid.width or new_h != grid.height: update grid.width, grid.height, and reallocate all three arrays (char_indices, fg_colors, bg_colors) with the new cell count. Use `vec![0; cell_count]` for char_indices, `vec![[0,0,0,255]; cell_count]` for colors.
   - Register this system in AsciiOutputPlugin::build in the Update schedule, BEFORE the test_pattern_system (so the pattern fills the newly-sized grid).
   - Import `bevy::window::WindowResized`.

2. **Verify prepare system handles dimension changes** in `engine-port/src/output/gpu_plugin.rs`:
   - The prepare system (from Plan 02) already checks `extracted.width != textures.last_width || extracted.height != textures.last_height` to recreate GPU textures. Confirm this is implemented. If not, add the dimension-change check and texture recreation logic.
   - When dimensions change: create new textures via render_device.create_texture() with new Extent3d, recreate the bind group with new texture views + existing font atlas view + uniform buffer.
   - After resize, GPU textures and bind group must be consistent (no stale references).

3. **Add resize unit test**: in a `#[cfg(test)] mod tests` block in mod.rs or in the existing test file, test that given a grid of 240x135 and a simulated resize to 1280x720 with font 10x16, the new dimensions are 128x45.
   - **I-04 NOTE:** This test covers 1:1 DPI (logical == physical pixels). See also `resize_retina_*` tests added per P3-M04 for HiDPI/Retina scenarios where physical dimensions differ from logical dimensions.
  </action>
