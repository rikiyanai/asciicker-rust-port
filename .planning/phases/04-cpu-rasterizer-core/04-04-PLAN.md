<objective>
Implement the RESOLVE stage and pipeline skeleton that produces AnsiCell output from rasterized SampleBuffer data.

Purpose: The RESOLVE stage is the final CPU step that converts the 2x-supersampled SampleBuffer into a grid of AnsiCell values. It reads 2x2 sample blocks, determines whether each cell is terrain (material lookup) or mesh (auto_mat lookup), applies elevation-based glyph selection, and handles grid/wireframe overlays. The pipeline stage enum provides the skeleton for the full 6-stage pipeline (stages 2-5 are stubs for Phase 5).

Output: Working resolve() function that produces correct AnsiCell output from test geometry, pipeline stage enum, and benchmark confirming 60fps+ at 240x135.
</objective>

  **resolve function** (port of C++ render.cpp:3412-3938, simplified for Phase 4):
  ```rust
  pub fn resolve(
      samples: &[Sample],
      dw: i32,              // sample buffer width (2*ascii_w + 4)
      dh: i32,              // sample buffer height (2*ascii_h + 4)
      ascii_w: i32,         // output width
      ascii_h: i32,         // output height
      materials: &[Material], // material library
      output: &mut [AnsiCell], // ascii_w * ascii_h output cells
  )
  ```

  **Per-cell logic** (for each output cell at (cx, cy)):
  1. Compute sample buffer position: `sx = 2 + 2*cx`, `sy = 2 + 2*cy` (skip the +2 border)
  2. Read 2x2 block: samples at `[sy*dw+sx, sy*dw+sx+1, (sy+1)*dw+sx, (sy+1)*dw+sx+1]`
  3. Average the 4 samples' height (ignoring CLEAR_HEIGHT samples), average diffuse
  4. Collect spare flags: OR of all 4 samples' spare bits
  5. Determine if mesh or material: `combined_spare & MESH_FLAG != 0` -> mesh path, else material path

  **Material path** (spare & 0x8 == 0):
  - Take the dominant visual (material index) from the 4 samples (use first non-clear sample)
  - Compute elevation (0-3) from row above: read `samples[(sy-1)*dw + sx]` height, compare to current block average height. If current > above: elevation increases. Map height difference to 0-3 range.
  - **P4-009 FIX:** Implementation uses elevation thresholds 0.5/2.0/5.0 height-difference breakpoints for 0-3 elevation mapping. Specifically: diff < 0.5 → elevation 0, diff < 2.0 → elevation 1, diff < 5.0 → elevation 2, diff >= 5.0 → elevation 3. These are approximate values that will need tuning with real terrain data in Phase 5 (see GAP-08).
  - **R6-M03 FIX:** Implementation reads only `src[-dw]` (row above) for elevation detection, matching C++ behavior. The RESEARCH.md Pitfall 5 mention of `src[dw]` (row below) was an error in the research analysis; only the row above is used for terrain elevation computation.
  - Compute diffuse level: `avg_diffuse / 17` clamped to 0-15
  - **P4-H03 FIX:** The `diffuse_level = avg_diffuse / 17` computation above is ONLY correct if you bypass `lookup()` and index `shade[elevation][diffuse_level]` directly. If you use `Material::lookup(elevation, diffuse)`, pass the RAW `avg_diffuse` (0-255) — `lookup()` already divides by 17 internally. DO NOT pre-divide and then pass to `lookup()` — that double-divides and collapses all diffuse variation to index 0. Simplest correct form: `lookup(elevation, avg_diffuse)`.
  - **R6-H03 FIX:** `auto_mat_lookup` unpacking specification: Given `rgb555: u16`, unpack as `r5 = rgb555 & 0x1F, g5 = (rgb555 >> 5) & 0x1F, b5 = (rgb555 >> 10) & 0x1F`. Index into AUTO_MAT: `idx = 3 * (r5 + 32*g5 + 32*32*b5)`. Implementation uses `lookup(elevation as u8, avg_diffuse)` without pre-dividing — the `lookup()` function handles the `/17` division internally.
  - Look up `materials[visual as usize].shade[elevation][diffuse_level]` -> MatCell
  - Convert MatCell.fg and MatCell.bg via rgb2pal -> fg_palette, bg_palette
  - Use MatCell.gl as the glyph
  - Write AnsiCell { fg: fg_palette, bk: bg_palette, gl, spare: 0xFF }

  **Mesh path** (spare & 0x8 != 0):
  - Take the dominant visual (RGB555) from the 4 samples (use first non-clear sample from [s00, s10, s01, s11] in reading order, same as material path)
  - **P4-L02 FIX:** "Dominant" means the first non-clear sample — not a majority vote. The material path description includes the "(use first non-clear sample)" clarification; the mesh path description was missing it. Added above.
  - Apply diffuse scaling: expand RGB555 to RGB888 via rgb5_to_rgb8, multiply by avg_diffuse/255, convert back to RGB555 via rgb8_to_rgb5
  - Look up auto_mat_lookup(scaled_rgb555) -> (bg, fg, dither_glyph)
  - Write AnsiCell { fg, bk: bg, gl: dither_glyph, spare: 0xFF }

  **Reflection path** (spare & PARITY_MASK == REFLECTION):
  - ~~Same as mesh/material but use diffuse divisor 400 instead of 255 (darker reflections, TRAP-R03)~~ **[WRONG — see P4-C01 FIX below]**
  - For Phase 4: implement the check but the actual reflection samples will only appear in Phase 5
  - **P4-C01 FIX (CRITICAL):** The ~~struck~~ description above is WRONG. Reflection does NOT apply to both mesh and material paths. The actual implementation sets `is_reflection` only when `(combined_spare & spare_bits::MESH_FLAG) == 0`, meaning reflection applies ONLY to the terrain/material path. For mesh samples (MESH_FLAG set), `is_reflection` is always false and `diffuse_divisor` stays 255 — reflection darkening is never applied to mesh samples. The correct description is: "Reflection path (spare & PARITY_MASK == REFLECTION AND spare & MESH_FLAG == 0): terrain/material path only. Mesh samples are never treated as reflections. The `diffuse_divisor=400` is passed to `resolve_mesh()` calls but the `is_reflection` guard requires `MESH_FLAG==0`, so in practice reflection darkening is a terrain-only feature as implemented."

  **Grid/wireframe overlay:**
  - If combined_spare & GRID (0x04): override glyph with grid character (b'+' for intersection, b'-' for horizontal, b'|' for vertical -- simplified for Phase 4)
  - If combined_spare & WIREFRAME (0x40): override glyph with wireframe character

  **Clear cells** (all 4 samples at CLEAR_HEIGHT):
  - Write AnsiCell { fg: 0, bk: 0, gl: b' ', spare: 0 }
  - **L-03 NOTE (Round 4):** `spare: 0` (NOT `0xFF`) for clear/sky cells. `0xFF` is the sentinel for "rendered cell" (material or mesh path). Clear cells are intentionally `spare: 0` to distinguish unrendered sky from rendered content. A reader expecting `0xFF` on all cells would be wrong; clear cells must not carry the rendered-cell sentinel.

  **Module registration** (engine-port/src/render/mod.rs):

  - Resolve with mixed terrain/mesh in 2x2 block: mesh flag dominates (matches C++ behavior)
  - Elevation detection: sample with height > row-above height produces elevation > 0
  - Output dimensions: ascii_w * ascii_h AnsiCells produced
  - **P4-H02 FIX:** IMPLEMENTATION NOTE: The GAP-11 test requirement (verify resolved AnsiCell is "darker" for reflection path) cannot be satisfied by this test list. The `diffuse_divisor=400` is only consumed by `resolve_mesh()`, not `resolve_material()`. For terrain/material path, the reflection check sets `is_reflection` but `diffuse_divisor` is not passed to `resolve_material()`, so there is no "darker output" to verify. The `test_resolve_material_reflection_path` test (added in Phase 3.1) verifies render completeness only (spare=0xFF, valid palette indices), not the "darker" property. The "darker output" property is only observable for mesh samples, but the `is_reflection` guard requires `MESH_FLAG==0`, making mesh+reflection mutually exclusive. This is a known semantic gap.
