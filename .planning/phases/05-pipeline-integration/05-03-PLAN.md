Create `camera.rs` with:
- `GameCamera` resource:
  ```rust
  #[derive(Resource)]
  pub struct GameCamera {
      // Input state
      pub pos: [f32; 3],          // world position
      pub yaw: f32,               // rotation in degrees
      pub zoom: f32,              // zoom level (default 1.0)
      pub perspective: bool,      // true = architectural perspective, false = isometric
      pub scene_shift: [i32; 2],  // screen shake offset (applied * 2 internally)

      // Derived state (recomputed each frame by update_camera)
      pub focal: f32,             // perspective focal length = max(dw,dh) * 2
      pub view_tm: [f64; 16],     // 4x4 view matrix (column-major)
      pub inv_tm: [f64; 16],      // inverse view matrix (for unprojection)
      pub view_dir: [f32; 3],     // normalized view direction (perspective only)
      pub view_pos: [f32; 3],     // camera world position (perspective only)
      pub view_ofs: [f32; 2],     // screen center offset (perspective only)
      pub frustum_planes: Vec<[f64; 4]>,  // extracted frustum planes [a,b,c,d] where ax+by+cz+d>=0 = inside
  }
  ```
- `GameCamera::default()` with sensible defaults: pos=[0,0,0], yaw=0, zoom=1.0, perspective=true, scene_shift=[0,0], zeroed derived state
- `GameCamera::update(&mut self, dw: f64, dh: f64)` method porting C++ render.cpp:2966-3034:
  - Compute yaw angle in radians: `a = yaw * PI / 180`
  - `sinyaw = a.sin()`, `cosyaw = a.cos()`
  - `scale = dh / dw.max(dh)`
  - `zoom_scaled = zoom as f64 * scale`
  - `sin30 = 0.5` (isometric constant)
  - `ds = 2.0 * zoom_scaled / VISUAL_CELLS as f64`
  - Build isometric view matrix (used for both modes):
    - `view_tm[0] = cosyaw * ds`
    - `view_tm[1] = -sinyaw * sin30 * ds`
    - `view_tm[4] = sinyaw * ds`
    - `view_tm[5] = cosyaw * sin30 * ds`
    - `view_tm[8] = 0.0`
    - `view_tm[9] = -ds`
    - Translation: `view_tm[12] = dw/2 - (pos[0]*view_tm[0]*HC + pos[1]*view_tm[4]*HC + pos[2]*view_tm[8]) + scene_shift[0]*2` where HC = HEIGHT_CELLS as f64
    - `view_tm[13] = dh/2 - (pos[0]*view_tm[1]*HC + pos[1]*view_tm[5]*HC + pos[2]*view_tm[9]) + scene_shift[1]*2`
    - CRITICAL (TRAP-R06): scene_shift values multiplied by 2 in both tm[12]/tm[13] and view_ofs
  - If perspective:
    - `focal = dw.max(dh) as f32 * 2.0`
    - `view_dir = [-sinyaw as f32, cosyaw as f32, 0.0]` (architectural perspective: no vertical tilt, sin30/cos30 commented out in C++)
    - `view_pos = [HC as f32 * pos[0] - view_dir[0] * focal, HC as f32 * pos[1] - view_dir[1] * focal, pos[2]]`
    - Normalize view_dir by focal: `view_dir[0] /= focal`, `view_dir[1] /= focal`
    - `view_ofs = [(dw/2 + scene_shift[0] as f64 * 2.0) as f32, (dh/2 + scene_shift[1] as f64 * 2.0) as f32]`
- `GameCamera::extract_frustum_planes(&mut self, dw: f64, dh: f64)`:
  - Extract 6 frustum planes from the view-projection matrix
  - For the isometric case: construct planes from the screen bounds transformed to world space
  - For the perspective case: construct planes from the camera position + view direction + FOV
  - Store in self.frustum_planes as Vec<[f64; 4]>
  - Each plane [a,b,c,d] is normalized such that ax+by+cz+d >= 0 means inside
  - Include near and far planes (near = 0, far = large value like 100000)
  - NOTE: The exact plane extraction math should match C++ QueryTerrain/QueryWorld expectations. The C++ passes plane arrays constructed in Render(). Port the exact construction from render.cpp.
- `GameCamera::frustum_planes(&self) -> &[[f64; 4]]` accessor that returns the plane slice

```rust
pub fn camera_input_system(
    keyboard: Res<ButtonInput<KeyCode>>,
    mut camera: ResMut<GameCamera>,
) {
    // Q/E rotation toggle
    if keyboard.just_pressed(KeyCode::KeyQ) {
        camera.yaw -= 45.0;  // or configurable increment
    }
    if keyboard.just_pressed(KeyCode::KeyE) {
        camera.yaw += 45.0;
    }
    // WASD movement (basic, can be refined later)
    let speed = 0.5;
    if keyboard.pressed(KeyCode::KeyW) {
        camera.pos[1] += speed;
    }
    if keyboard.pressed(KeyCode::KeyS) {
        camera.pos[1] -= speed;
    }
    if keyboard.pressed(KeyCode::KeyA) {
        camera.pos[0] -= speed;
    }
    if keyboard.pressed(KeyCode::KeyD) {
        camera.pos[0] += speed;
    }
}
```

```rust
}
```

Create `shadow.rs` with:
- `update_terrain_dark` function porting C++ terrain.cpp:1714-1765 (DarkUpdater):
  ```rust
  pub fn update_terrain_dark(
      terrain: &mut RuntimeTerrain,
      light_pos: [f32; 3],
  )
  ```
  Implementation:
  - Compute light direction: `light_dir = [-light_pos[0] as f64, -light_pos[1] as f64, -light_pos[2] as f64 * HEIGHT_SCALE as f64]`
  - Use `terrain.for_each_patch_mut()` to iterate all patches:
    - For each patch, compute dark bitmask:
    - For v in 0..VISUAL_CELLS, for u in 0..VISUAL_CELLS:
      - Compute cell center world coordinates via `patch.sample_cell_center(u, v, px, py)`
      - Bit index: `bit = 1u64 << (u + VISUAL_CELLS * v)`
      - Raycast from cell center in light_dir direction to test terrain self-shadowing
      - If hit terrain at height > cell_z + HEIGHT_SCALE/4: set dark bit
      - Note: Full world geometry raycasting (HitWorld) is complex and can be deferred. For Phase 5, implement terrain self-shadowing only. World-based shadowing requires the BSP raycast (HitWorld equivalent) which is Phase 5 Plan 02 scope -- the wiring can happen in Plan 04.
    - Write `patch.dark = dark`

- `terrain_raycast_height` helper function for simplified terrain self-shadow:
  ```rust
  fn terrain_raycast_height(
      terrain: &RuntimeTerrain,
      origin: &[f64; 3],
      direction: &[f64; 3],
      max_steps: usize,
  ) -> Option<f64>
  ```
  Implementation (simplified for Phase 5):
  - March along the ray in discrete steps (step size = 1 patch unit)
  - At each step, compute x,y position, find the patch at that position
  - If patch exists, interpolate terrain height at that x,y position
  - If ray height < terrain height at any step, return Some(terrain_height)
  - After max_steps, return None (no shadow)
  - NOTE: This is a simplified version of C++ HitTerrain. The full ray-triangle intersection can be refined later. The key is producing a reasonable dark bitmask for visual correctness.
