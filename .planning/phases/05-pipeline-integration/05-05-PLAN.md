  **P5-321 FIX (LOW) — SpriteQueue serialization performance constraint:**
  `SpriteQueue` is accessed as `ResMut<SpriteQueue>` by BOTH `query_character_sprites`
  (Phase 6, CharacterPlugin) and `render_pipeline_system` (Phase 5, CpuRasterizerPlugin).
  Because both systems take mutable access to the same Resource, Bevy's scheduler CANNOT
  run them in parallel — they are forcibly serialized.

  This means the entire character sprite query and the entire render pipeline become
  sequential. This is ACCEPTABLE for the current design because:
  1. Both systems MUST be ordered anyway (sprites must be queued before pipeline reads them).
  2. The serialization overhead is zero (they would be sequential regardless of .after()).
  3. SpriteQueue push is O(n) where n = visible characters (typically < 50).

  If profiling shows this is a bottleneck (unlikely), consider:
