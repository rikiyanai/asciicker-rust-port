- `LIGHT_DIR_DEFAULT: [f64; 3]` constant:
  - Default light direction: `[-1.0, 1.0, -2.0]` (normalized, pointing from upper-right)
  - The C++ code computes light_dir from `light_pos` with Z scaled by HEIGHT_SCALE. We hardcode a reasonable default for Phase 5.

- `update_terrain_dark` function:
  ```rust
  pub fn update_terrain_dark(
      terrain: &mut RuntimeTerrain,
      light_dir: [f64; 3],
  )
  ```
  Implementation (port of C++ terrain.cpp:1714-1765 DarkUpdater):
  **P5-058 FIX: BORROW CONFLICT:** `for_each_patch_mut(&mut self)` holds a mutable borrow. Calling `self.interpolate_height()` inside the closure requires an immutable borrow — this won't compile. Fix: use index-based iteration instead of closure-based `for_each_patch_mut`, or collect height data into a local snapshot before the mutable pass.
  1. Two-pass approach to avoid borrow conflict:
     - **Pass 1**: Collect all shadow results. For each patch, iterate all 64 visual cells:
       a. Compute world-space cell center via `patch.sample_cell_center(u, v, px, py)`
          **P5-315 FIX (MEDIUM):** In the two-pass approach, the shadow computation iterates patches by index (not via `for_each_patch_mut`). The `px` and `py` values are NOT parameters of the closure — they must be read from the `RuntimePatch` fields: `patch.x` and `patch.y`. The correct call is: `patch.sample_cell_center(u, v, patch.x, patch.y)`. Do NOT pass placeholder values or reuse a loop variable for `px`/`py`. The `RuntimePatch` struct (Plan 05-01) stores `x: i32, y: i32` for this purpose.
       b. Cast a ray from cell center along `light_dir`
       c. Test terrain self-shadowing: `terrain_raycast_height` walks along ray, interpolating terrain height at each step
       d. If terrain height at step > ray height at step (with HEIGHT_SCALE/4.0 tolerance per C++), mark cell as shadowed
       e. Collect (patch_index, dark_bitmask) pairs
     - **Pass 2**: Write dark values to patches
  2. Write `patch.dark = dark` for each patch

  NOTE: Full world geometry shadowing (HitWorld) is deferred to Phase 6. Phase 5 implements terrain self-shadowing only. This is documented and acceptable because:
  - Terrain self-shadow is the dominant shadow contributor in most scenes
  - World shadow requires mesh raycast infrastructure not yet available
  - The dark bitmask format supports future world shadow additions (just OR in more bits)

- `terrain_raycast_height` helper:
  ```rust
  fn terrain_raycast_height(
      terrain: &RuntimeTerrain,
      origin: &[f64; 3],
      dir: &[f64; 3],
      max_steps: usize,
  ) -> bool
  ```
  Implementation:
  - Step along ray from origin in direction dir with step size = 1.0 (visual cell spacing)
    **P5-311 FIX (MEDIUM):** The step size of 1.0 assumes world coordinates where each visual cell occupies 1.0 unit in X and Y. Verify this against `sample_cell_center` output units. The C++ HEIGHT_SCALE constant applies to Z only; X and Y are in direct visual-cell units (each visual cell is 1 unit). If HEIGHT_CELLS=4 and VISUAL_CELLS=8, there are 2 visual cells per height cell, so a step of 1.0 visual-cell unit is correct for shadow resolution. However, a step of 2.0 (one height-cell unit) may be needed to align with height vertex spacing. Confirm with: `assert_eq!(VISUAL_CELLS / HEIGHT_CELLS, 2)` and use step size 1.0 for shadow quality (over-samples by 2x relative to height grid, which is acceptable).
  - At each step, compute world (x, y) position and interpolate terrain height via `terrain.interpolate_height(wx, wy)`
    **P5-120 FIX (MEDIUM):** `interpolate_height` returns `Option<f64>` (None when outside
    all terrain patches). Handle the None case with a `let-else` continue — outside terrain
    bounds means no terrain to shadow from, so skip this ray step:
    ```rust
    let Some(terrain_z) = terrain.interpolate_height(wx, wy) else { continue; };
    ```
    Do NOT use `.unwrap()` — rays that exit the terrain bounds will panic.
  - If interpolated terrain height > ray z at that step (with HEIGHT_SCALE/4.0 tolerance per C++), return true (shadowed)
  - After max_steps, return false (not shadowed)
  - max_steps default: 32 (sufficient for typical terrain extents)
  **R7-003 FIX:** Explicit call pattern in the Pass 1 per-cell loop:
  ```rust
  if terrain_raycast_height(&terrain, &cell_center, &light_dir, 32) {
      dark |= 1u64 << cell_bit;
  }
  ```
  The tolerance check (`terrain_height > ray_height + HEIGHT_SCALE as f64 / 4.0`) is INSIDE `terrain_raycast_height` (returns true if shadowed). The bit-setting is in the outer Pass 1 loop.
  **P5-330 FIX:** HEIGHT_SCALE = 256 (from engine-port/src/asset_loader/constants.rs — verify before use). Both `interpolate_height` return values and ray Z are in raw height units (u16 as f64). The `light_dir[2]` must be scaled by HEIGHT_SCALE: `light_dir_z_scaled = LIGHT_DIR_DEFAULT[2] * HEIGHT_SCALE as f64`. The tolerance check: `terrain_height > ray_z + HEIGHT_SCALE as f64 / 4.0`.
