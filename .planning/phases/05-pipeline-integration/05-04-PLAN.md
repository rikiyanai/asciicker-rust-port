### 1B: TerrainShader struct (`render/terrain_shader.rs`)

Create `terrain_shader.rs` implementing the C++ RenderPatch equivalent (~200 lines in C++). This is the core terrain rendering logic that connects Phase 4's rasterizer to Phase 5's terrain data.

**TerrainShader struct** implementing `RasterShader`:
```rust
pub struct TerrainShader {
    pub material_index: u16,  // visual[vy][vx] from the terrain patch
    pub diffuse_base: u8,     // base diffuse lighting (can be modulated by normal)
}
```

**`blend(&self, sample: &mut Sample, z: f32, _bc: [f32; 3])`:**
- Depth test: `if sample.height > z || sample.height == Sample::CLEAR_HEIGHT`
  - Depth convention: SMALLER z = closer to camera (wins depth test). A near triangle at z=50 overwrites a far triangle at z=200 because `sample.height (200) > z (50) = TRUE`.
  - Use the inline `height > z || CLEAR_HEIGHT` pattern. Do NOT use `depth_test_ro()` in shader blends -- it has a semantic inversion (returns TRUE when existing sample wins, i.e., new fragment does NOT overwrite) and uses HALF_HEIGHT_SCALE epsilon tolerance. The inline pattern guards the WRITE path (condition true = new fragment wins). Both terrain and mesh shaders MUST use the same inline pattern.
  - Note: `depth_test_ro()` IS correctly used in `bresenham()` for grid lines -- that epsilon tolerance is deliberate for line painting. Do not change bresenham.
- Write: `sample.visual = self.material_index` (terrain uses material index, NOT RGB555)
- Write: `sample.diffuse = self.diffuse_base`
- Write: `sample.spare = 0` (NO MESH_FLAG -- this is terrain)
- Write: `sample.height = z`

**Critical: spare bit behavior.**
- spare bit 3 (MESH_FLAG) must be 0 for terrain. The resolve pass branches on this.
- `Sample::clear_state()` initializes `spare = spare_bits::MESH_FLAG`. All initially-cleared samples have MESH_FLAG set. Do NOT add a guard like `if sample.spare & MESH_FLAG == 0` before writing terrain -- this would skip all uninitialized (clear) terrain regions. The CLEAR_HEIGHT check in the depth test is the only correct gate. The `sample.spare = 0` write overrides the MESH_FLAG from clear_state, which is intended.
- In 2x2 blocks where terrain only partially covers the buffer (patch edges), combined_spare in resolve() ORs in MESH_FLAG from adjacent clear samples, causing edge blocks to use the mesh resolve path. This is expected behavior consistent with Phase 4 design.

### 1C: render_patch function

The core terrain-to-rasterizer bridge. Callers must drop the `&mut [Sample]` borrow before calling `resolve_to_grid(&SampleBuffer, ...)`. Use `{}` scoping blocks to explicitly end the mutable slice borrow.

```rust
/// buf_w, buf_h: SAMPLE buffer dimensions (2*ascii_w+4, 2*ascii_h+4), NOT ASCII dimensions
pub fn render_patch(
    buf: &mut [Sample],
    buf_w: i32,
    buf_h: i32,
    patch: &RuntimePatch,
    patch_x: i32,
    patch_y: i32,
    view_tm: &[f64; 16],
)
```

**Implementation (port of C++ RenderPatch):**
- The patch has HEIGHT_CELLS=4 quads per edge (5x5 vertices)
- For each quad (hx, hy) in 0..HEIGHT_CELLS:
  - Get 4 corner heights from `patch.height[hy][hx]`, `[hy][hx+1]`, `[hy+1][hx]`, `[hy+1][hx+1]`
  - Compute world-space vertex positions: each height cell spans (VISUAL_CELLS/HEIGHT_CELLS = 2) visual cells
  - World X = patch_x * VISUAL_CELLS + hx * (VISUAL_CELLS/HEIGHT_CELLS)
  - World Y = patch_y * VISUAL_CELLS + hy * (VISUAL_CELLS/HEIGHT_CELLS)
  - World Z = height_value (raw u16, HEIGHT_SCALE applied later in projection)
  - Transform each vertex through view_tm via `transform_vertex` (imported from `crate::render::math`) to get sample-buffer [x, y, z] integer coords
  - Look up material from `patch.visual[vy][vx]` where vy/vx map from the height cell to the visual grid (scale factor 2). Array is row-major: `visual[row][col]` = `visual[vy][vx]` where vy is Y index (0..8) and vx is X index (0..8). Do NOT use `visual[vx][vy]` -- that transposes the material layout.
  - Read `patch.dark` bitmask to modulate diffuse lighting for shadowed cells:
    ```rust
    let cell_bit = u + v * VISUAL_CELLS; // visual cell index within patch (0..63), VISUAL_CELLS = 8
    let is_shadowed = (patch.dark >> cell_bit) & 1 != 0;
    let diffuse = if is_shadowed { diffuse_base / 2 } else { diffuse_base };
    ```
    This halves diffuse for shadowed cells, matching C++ behavior.
  - Check `patch.diag` bit to determine triangle split direction for this quad
  - Create TerrainShader with the material_index and computed diffuse value
  - Call `rasterize(buf, buf_w, buf_h, &shader, [&v0, &v1, &v2], false)` for each triangle (2 per quad)
  - Also call `bresenham()` for grid lines if grid rendering is enabled (spare_bits::GRID)

/// Trait for pluggable glyph selection at RESOLVE stage.

### 2D: resolve_to_grid function (`render/resolve_bridge.rs`)

**CRITICAL GAP #3 RESOLUTION**: Bridges AnsiCell (xterm-256 palette u8) to AsciiCellGrid (RGBA [u8; 4]).

**`XTERM_256_PALETTE: [[u8; 3]; 256]` constant:** Full xterm-256 palette RGB values.
- Indices 0-15: standard terminal colors. Index 0 maps to [0,0,0] (terminal default black).
- Indices 16-231: 6x6x6 color cube: `r = (idx - 16) / 36 * 51`, `g = ((idx - 16) % 36) / 6 * 51`, `b = (idx - 16) % 6 * 51`
- Indices 232-255: grayscale ramp: `v = 8 + (idx - 232) * 10`
- The formula uses evenly-spaced levels [0,51,102,153,204,255]. This is INTENTIONAL -- the C++ engine's `rgb2pal()` uses `(c + 25) / 51` which assumes evenly-spaced levels. The palette table must be the inverse of `rgb2pal` for correct round-tripping.

**Canonical signature** (use GENERIC form -- all consumers must match this):
```rust
pub fn resolve_to_grid<G: GlyphSelector>(
    sample_buffer: &SampleBuffer,
    materials: &[Material],
    grid: &mut AsciiCellGrid,
    glyph_selector: &mut G,
    resolve_buf: &mut Vec<AnsiCell>,
)
```

Plans 05-05, 06-03, and 07-04 MUST call with all 5 parameters.
Default callers pass `&mut AutoMatGlyphSelector` and `&mut Vec::new()` (or a pre-allocated buffer).
The `resolve_buf` avoids re-allocating the `Vec<AnsiCell>` every frame.
For unit tests, pass `&mut Vec::new()` directly as the resolve_buf.

**Implementation:**

Derive internal params from SampleBuffer fields (`width` and `height` are both `pub` fields, not methods):
```rust
let dw = sample_buffer.width as i32;  // sample width (e.g. 484)
let dh = sample_buffer.height as i32; // sample height (e.g. 274)
let ascii_w = (sample_buffer.width as i32 - 4) / 2;  // e.g. 240
let ascii_h = (sample_buffer.height as i32 - 4) / 2;  // e.g. 135
```
Verify in sample_buffer.rs that both `pub width: u32` and `pub height: u32` are struct fields before writing code that accesses them.

Validate grid dimensions match before indexing:
```rust
assert_eq!(grid.width as i32, ascii_w, "AsciiCellGrid width mismatch");
assert_eq!(grid.height as i32, ascii_h, "AsciiCellGrid height mismatch");
```

1. Resize `resolve_buf` to EXACTLY the target size using `Vec::resize()` (NOT `reserve()`):
   `resolve_buf.resize((ascii_w * ascii_h) as usize, AnsiCell::default());`
   `resolve()` has `debug_assert_eq!(output.len(), (ascii_w * ascii_h) as usize)` that panics on mismatch. Always resize to exactly the target -- `Vec::resize` handles both growth and shrinkage.

2. Call Phase 4's resolve:
   `resolve(&sample_buffer.samples, dw, dh, ascii_w, ascii_h, materials, resolve_buf)`
   Inside `resolve_to_grid`, the `materials` parameter is already `&[Material]` -- do NOT write `materials.0` here. The `.0` unwrap of `RuntimeMaterials` happens at the CALL SITE in pipeline.rs (05-05), not inside this function.

3. For each AnsiCell in the output:
   - AnsiCell has exactly 4 fields: `fg: u8`, `bk: u8`, `gl: u8`, `spare: u8`. There is NO `flags` field.
   - First, call `glyph_selector.select_glyph(sample_buffer, cell_x, cell_y)`
   - If `Some(glyph)`, override `cell.gl = glyph` (keep auto_mat fg/bg colors)
   - If `None`, use auto_mat glyph as-is
   - Convert `cell.fg` (xterm-256 index) to RGBA via `XTERM_256_PALETTE[cell.fg as usize]` + alpha=255
   - Convert `cell.bk` (xterm-256 index) to RGBA via `XTERM_256_PALETTE[cell.bk as usize]` + alpha=255
   - Write to grid: `grid.char_indices[i] = cell.gl as u16`, `grid.fg_colors[i] = [r, g, b, 255]`, `grid.bg_colors[i] = [r, g, b, 255]`

4. Handle clear cells: write glyph=32 (space, matching resolve()'s `gl: b' '`), fg=[0,0,0,255], bg=[0,0,0,255]
   - AnsiCell.spare is an internal pipeline flag (0xFF = rendered, 0 = clear). It is NOT written to AsciiCellGrid. To detect clear cells downstream, check AsciiCellGrid: `char_indices[i] == 32` AND `fg_colors[i] == [0,0,0,255]` AND `bg_colors[i] == [0,0,0,255]`.

**P5-302 (CRITICAL):** The reason to avoid `depth_test_ro()` is primarily a SEMANTIC INVERSION. `depth_test_ro()` returns TRUE when EXISTING sample wins (new fragment does NOT overwrite). The inline pattern guards the WRITE path (condition true = new fragment wins). Using `depth_test_ro()` without negation would silently SKIP all fragment writes. To use correctly: `if !sample.depth_test_ro(z) { write fragment }`. Error-prone -- stick with inline.

**P5-320 (INFO):** `depth_test_ro()` IS correctly used in `bresenham()` (rasterizer.rs line ~264): `if buf[idx].depth_test_ro(z) { buf[idx].spare |= or_bits; }`. This is correct because bresenham ORs grid bits when existing sample does NOT block the line. The epsilon tolerance is intentional for line painting.

**XP-206 (LOW):** P5-055's original claim was not marked WRONG inline until this fix annotated it.

### MESH_FLAG and clear_state Interaction (P5-106, P5-116, TRAP-R01)
