**state_machine.rs:** Port ActionState enum from game.h ACTION, adding Block state per CHAR-01 requirement ("idle, walk, run, attack, block, dead" -- 6 states).

Variants: None (idle/walk/run -- speed determined by PhysicsIO velocity magnitude), Attack, Block, Fall, Dead, Stand (standing up from fall).

NOTE on Block: The C++ engine does not have a separate ACTION_BLOCK enum value -- blocking is handled via shield equipment state + input flag. For the Rust port, we add an explicit Block variant to ActionState per the CHAR-01 requirement. Block behaves like a movement-locked state similar to Attack: character cannot walk or run while blocking.

Implement `can_transition_to(&self, target) -> bool` with C++ guards + Block rules:
- None: can always transition to
- Attack: blocked from Fall, Stand, Dead, Block
- Block: blocked from Fall, Stand, Dead, Attack (cannot block while attacking or vice versa)
- Fall: blocked from Dead
- Stand: only from Fall or Dead
- Dead: can transition to from any state

Method `try_transition(&mut self, target) -> bool`: checks guard, applies if valid, returns success.

Method `is_movement_locked(&self) -> bool`: returns true for Attack, Block, Dead, Stand, Fall -- states where the character cannot voluntarily move.

**sprite_query.rs:** System that bridges character state to the rendering pipeline's deferred sprite blit queue.

```rust
/// Reads character entities with SpriteReq + AnimationState + Transform,
/// creates SpriteRenderEntry items and pushes them to the SpriteQueue resource.
/// Runs after physics and before the render pipeline.
pub fn query_character_sprites(
    characters: Query<(&SpriteReq, &AnimationState, &Transform)>,
    camera: Res<GameCamera>,
    mut sprite_queue: ResMut<SpriteQueue>,
) {
    for (sprite_req, anim_state, transform) in &characters {
        // 1. Determine which XP sprite to reference from SpriteReq:
        //    - sprite_req.kind determines sprite family (Human/Wolf/Bee)
        //    - sprite_req.action determines action variant (idle/attack/fall/etc)
        //    - sprite_req.weapon/shield/helmet/armor determine equipment variant
        //    - Combine into a sprite_name string: "{kind}_{action}_{equipment_hash}"
        //    NOTE: Full sprite name resolution requires the Phase 2 sprite
        //    asset naming convention. For now, construct a canonical name.

        // 2. Get current animation frame from anim_state.current_frame

        // 3. Compute screen position by projecting transform.translation
        //    through camera.view_tm

        // 4. Compute distance from camera for far-to-near sort

        // 5. Create SpriteRenderEntry and push to sprite_queue
        let pos = transform.translation;
        let entry = SpriteRenderEntry {
            dist: /* compute from camera */,
            screen_x: /* project through view_tm */,
            screen_y: /* project through view_tm */,
            sprite_name: /* from sprite_req */,
            pos: [pos.x, pos.y, pos.z],
            yaw: /* from PhysicsIO or transform rotation */,
            anim: /* action index */,
            frame: anim_state.current_frame as i32,
        };
        sprite_queue.push(entry);
    }
}
```

This system is the missing link (Gap 5) between character components and the render pipeline. Without it, character entities exist in ECS but are never drawn.
