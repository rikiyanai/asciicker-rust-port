PhysicsGeometrySource trait:
```rust
pub trait PhysicsGeometrySource {
    /// Collect triangles near `center` within `radius` (world-space).
    /// Triangles must be transformed to sphere space using mul_xy and mul_z
    /// before being pushed to soup.
    fn collect_triangles(
        &self,
        center: &[f32; 3],
        radius: f32,
        mul_xy: f32,
        mul_z: f32,
        soup: &mut Vec<SoupItem>,
    );
}
```

NOTE: The trait uses a spatial query (center + radius), NOT frustum planes. This is because physics needs geometry near the entity regardless of camera view direction. This differs from the frustum-based queries in Phase 5's rendering pipeline.

1. **TerrainGeometrySource:**
```rust
pub struct TerrainGeometrySource<'a> {
    pub terrain: &'a RuntimeTerrain,
}

impl<'a> PhysicsGeometrySource for TerrainGeometrySource<'a> {
    fn collect_triangles(
        &self,
        center: &[f32; 3],
        radius: f32,
        mul_xy: f32,
        mul_z: f32,
        soup: &mut Vec<SoupItem>,
    ) {
        // Port from C++ physics.cpp PatchCollect callback (lines 187-250):
        // 1. Compute patch coordinate range around center:
        //    px_min = ((center[0] - radius) / VISUAL_CELLS).floor() as i32
        //    px_max = ((center[0] + radius) / VISUAL_CELLS).ceil() as i32
        //    (same for y using center[1])
        // 2. For each patch in range, use terrain.get_patch_at(px, py)
        // 3. For each patch found, triangulate the HEIGHT_CELLS x HEIGHT_CELLS
        //    height grid into 2 triangles per cell (using patch.diag for orientation):
        //    - Each height cell spans (VISUAL_CELLS / HEIGHT_CELLS) visual cells
        //    - Vertex positions: world_x = px * VISUAL_CELLS + col * cell_size
        //                        world_y = py * VISUAL_CELLS + row * cell_size
        //                        world_z = patch.height[row][col] as f32 / HEIGHT_SCALE
        //    - Transform each vertex to sphere space via to_sphere_space
        //    - Compute triangle normal from cross product
        //    - Push SoupItem for each triangle
        // 4. Use patch.diag bitfield to determine triangle diagonal direction
        //    per height cell (bit = row * HEIGHT_CELLS + col)
    }
}
```

2. **WorldGeometrySource:**
```rust
pub struct WorldGeometrySource<'a> {
    pub world: &'a RuntimeWorld,
}

impl<'a> PhysicsGeometrySource for WorldGeometrySource<'a> {
    fn collect_triangles(
        &self,
        center: &[f32; 3],
        radius: f32,
        mul_xy: f32,
        mul_z: f32,
        soup: &mut Vec<SoupItem>,
    ) {
        // Port from C++ physics.cpp MeshCollect callback (lines 251-350):
        // 1. Walk BSP tree looking for mesh instances whose bbox
        //    intersects a sphere of (center, radius) in world space.
        //    Use RuntimeWorld's instances list. For each RuntimeInstance::Mesh:
        //    - Check if bbox overlaps with sphere: for each axis,
        //      closest point on AABB to center is within radius
        // 2. For each nearby mesh instance:
        //    - The instance has a transform matrix (tm). Mesh triangle data
        //      is not yet available as a runtime resource (Phase 5 loads meshes
        //      but full triangle access depends on AkmMesh triangle data being
        //      stored in RuntimeInstance or a separate MeshTriangleCache).
        //    - INITIAL IMPLEMENTATION: Use a simplified bounding-box collision
        //      proxy. Generate 12 triangles from the instance's bbox (2 per face
        //      of the AABB). This provides basic collision against world objects.
        //    - Transform bbox triangles to sphere space via to_sphere_space
        //    - Compute normals, push SoupItems
        //    - TODO: When AkmMesh triangle data is accessible at runtime,
        //      replace bbox proxy with actual mesh triangles transformed by tm
    }
}
```

NOTE on WorldGeometrySource: The C++ MeshCollect accesses the full mesh triangle data via Mesh* pointer. In Phase 5, RuntimeInstance stores the bbox but not individual mesh triangles. Using the bbox as a collision proxy is a pragmatic approach that gives correct collision behavior for axis-aligned geometry. Mesh-accurate collision is a Phase 7 refinement.
