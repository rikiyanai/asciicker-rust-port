### PhysicsIO (definitive field list)

```rust
#[derive(Resource)]
pub struct PhysicsIO {
    // INPUT (game -> physics): written by input system, read by physics
    pub x_force: f32,
    pub y_force: f32,
    pub z_force: f32,
    pub torque: f32,
    pub water: f32,           // water surface Z; NEG_INFINITY = no water

    // I/O (bidirectional):
    pub jump: bool,           // set by input, cleared by physics after evaluation
    pub fly: bool,
    pub x_impulse: f32,
    pub y_impulse: f32,

    // OUTPUT (physics -> game): written by physics, read by game/character
    pub pos: [f32; 3],
    pub yaw: f32,             // written by apply_torque_to_camera (Update), NOT by physics
    pub player_dir: i32,
    pub player_stp: i32,
    pub dt: f32,
    pub grounded: bool,
    pub vel_z: f32,           // vertical velocity output for character state transitions
    pub world_radius: f32,
    pub world_height: f32,
}
```

Key constraints:
- `vel_z` is written by `update_output_system` as `physics_state.vel[2]` each FixedUpdate tick. External systems MUST NOT write it. Plan 06-02 uses it to distinguish Jump (vel_z > 0) from Fall (vel_z < 0).
- `yaw` is NOT set by physics. `apply_torque_to_camera` (Update schedule) writes `physics_io.yaw = camera.yaw`.
- `jump` must be cleared to `false` AFTER physics evaluates it (prevents multi-substep re-reads with MAX_SUBSTEPS=10).
- `world_radius` and `world_height` are not in C++ PhysicsIO (computed locally in Animate()). Rust stores them in PhysicsIO for Bevy system parameter access.

**Unified gravity/buoyancy formula** (physics.cpp:1488-1539) -- NO separate `vel_z -= 9.8*dt`:
```rust
let h = io.world_height;
let cnt: f32 = 0.78;  // TRAP-P04
let acc = (io.water - (io.pos[2] + cnt * h)) / (2.0 * cnt * h);
let acc = acc.clamp(-cnt, 1.0 - cnt);

```
When water=NEG_INFINITY, `acc` clamps to `-cnt` (~-0.78), producing gravity. No discrete 9.8 constant exists in C++. Adding a separate gravity term would DOUBLE-COUNT.

### geometry.rs -- Free functions (no trait)

Two static geometry sources do not justify a trait. Free functions take the resource directly.

```rust
/// Collect terrain collision triangles near `center` within `radius` (world-space).
/// Transforms vertices to sphere space. Sets material from terrain vmap.
pub fn collect_terrain_triangles(
    terrain: &RuntimeTerrain,
    center: &[f32; 3],
    radius: f32,
    mul_xy: f32,
    mul_z: f32,
    soup: &mut Vec<SoupItem>,
)
```

Port from C++ physics.cpp PatchCollect (lines 187-250):
1. Compute patch coordinate range. Each patch covers VISUAL_CELLS (8) world units:
   `px_min = ((center[0] - radius) / VISUAL_CELLS as f32).floor() as i32` (same for y). All VISUAL_CELLS arithmetic requires explicit `as f32` casts.
2. For each patch in range, call `terrain.get_patch_at(px, py)`.
3. Triangulate height grid: (HEIGHT_CELLS+1)x(HEIGHT_CELLS+1) = 5x5 vertices producing 4x4 quads = 32 triangles per patch. Access heights as `patch.height[row * (HEIGHT_CELLS + 1) + col]`.
   - Vertex stepping: `sxy = VISUAL_CELLS as f32 / HEIGHT_CELLS as f32` = 2.0
   - Vertex positions: `world_x = px * VISUAL_CELLS as f32 + col * sxy`, `world_y = py * VISUAL_CELLS as f32 + row * sxy`
   - Height: `world_z = patch.height[row * (HEIGHT_CELLS + 1) + col] as f32 / HEIGHT_SCALE as f32` (HEIGHT_SCALE=16; raw 16 -> 1.0 world unit)
   - Transform each vertex to sphere space via `to_sphere_space`
   - Compute triangle normal from cross product, push SoupItem
4. Use `patch.diag` bitfield for triangle diagonal direction per cell (bit = row * HEIGHT_CELLS + col).
