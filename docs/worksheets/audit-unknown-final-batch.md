## 1. URDO Undo/Redo Format

**Source:** `/Users/rikihernandez/Downloads/Aciicker-Y9-2/urdo.cpp` (lines 1-78, header comments)

### Architecture: Doubly-Linked List with Cursor

```
[op1] <-> [op2] <-> [op3] <-> [op4] <-> [op5]
                      ^undo     ^redo
```

- **undo pointer:** Last executed operation (can be undone)
- **redo pointer:** Next undone operation (can be redone)
- **New operations:** Appended after undo pointer, redo chain is purged

### Group Nesting (Stack-Based, Up to 64 Levels)

- `URDO_Open()`: Push new group onto stack, start collecting operations
- `URDO_Close()`: Pop group, seal as single undo unit
- Example: Merge operation creates group containing:
  - `[GROUP] -> [PATCH_CREATE] -> [PATCH_UPDATE_HEIGHT] -> [PATCH_UPDATE_VISUAL]`
- Undo reverses ALL operations in group atomically
- Groups can nest (painting multiple patches creates outer group with inner groups)

### Six Operation Types

| Type | Description | Data Stored |
|------|-------------|-------------|
| `CMD_GROUP` | Nested group (undone/redone atomically) | `group_head`, `group_tail` pointers |
| `CMD_PATCH_CREATE` | Create/delete terrain patch (toggle attached state) | `terrain`, `patch`, `cx`, `cy`, `attached` |
| `CMD_PATCH_UPDATE_HEIGHT` | Snapshot of height map before editing (SWAP) | `patch`, `height[]`, `diag` |
| `CMD_PATCH_UPDATE_VISUAL` | Snapshot of visual map before editing (SWAP) | `patch`, `visual[][]` |
| `CMD_PATCH_DIAG` | Snapshot of diagonal flag before flipping (SWAP) | `patch`, `diag` |
| `CMD_INST_CREATE` | Create/delete mesh/sprite instance (toggle attached) | `inst`, `attached` |

### The SWAP Pattern (Critical Invariant)

The key insight is that `URDO_PatchUpdateHeight::Do()` and `URDO_PatchUpdateVisual::Do()` perform a **swap** rather than a copy:

```cpp
// After swap, URDO struct holds NEW data (becomes OLD on next undo/redo)
// Terrain holds restored OLD data
// Undo and redo are IDENTICAL operations - just swap back and forth
```

- **Why SWAP not copy:** O(1) operation (no allocation), minimal memory (single snapshot), undo and redo are symmetric
- **Height array size:** `uint16_t height[HEIGHT_CELLS + 1][HEIGHT_CELLS + 1]`
- **Visual array size:** `uint16_t visual[VISUAL_CELLS][VISUAL_CELLS]`

### Memory Tracking

- `bytes` counter: Total memory used by undo/redo history
- Per-operation size tracking: Each `Free()` decrements bytes
- Detached patch data: `TerrainDispose()` returns size when freeing
- Exposed via `URDO_Bytes()` for UI display

## 2. Font Used for Alphabet

**Source:** `/Users/r/Projects/ascii research/alexharri-ascii/` (Alex Harri's ASCII renderer research archive)

### Font: Fira Code

The alphabet character shape vectors were generated using **Fira Code** monospace font.

### Evidence

1. **From `cssVariables.ts`:**
   ```typescript
   export const cssVariables = {
     firaCode: "'Fira Code', monospace",
     // ...
   };
   ```

2. **From research documentation (`docs/worksheets/brainstorms/2026-02-17-ascii-effect-mix-project-brainstorm.md`):**
   - Resolved question: "**Font choice**: **Fira Code** — matches the precomputed alphabet shape vectors. Best character-shape fidelity."

3. **From ASCII renderer analysis:**
   - "Font is set to Fira Code (monospace)"
   - "Each glyph is a tiny canvas exactly the size of one grid cell. The font used is always Fira Code"

### Why Fira Code?

The precomputed 6D shape vectors in the alphabet JSON files were generated by sampling Fira Code glyphs. Using a different font would result in mismatched character shapes because each character has different geometric properties.

### Note on MonoLisa

The website also uses **MonoLisa** (for general UI text), but this is NOT the font used for the ASCII alphabet. The ASCII renderer specifically uses Fira Code.

## 3. stb_vorbis Error Handling

**Source:** `/Users/rikihernandez/Downloads/Aciicker-Y9-2/audio.cpp` (lines 218-231)

### Error Handling Implementation

```cpp
extern "C" void XOgg(int index, const uint8_t* data, int ogg_size)
{
    if (index>=MAX_SAMPLES)
        return;

    int err = 0;
    // Open Ogg stream from memory
    stb_vorbis* ogg = stb_vorbis_open_memory(data, ogg_size, &err, 0);
    if (!ogg)
    {
        lib_sample_data[index]=0;
        lib_sample_len[index]=0;
        return;
    }
    // ... continue decoding ...
}
```

### Behavior on Error

| Error Condition | Handling |
|-----------------|----------|
| `stb_vorbis_open_memory()` returns NULL | Silent failure |
| `lib_sample_data[index]` | Set to `0` (null pointer) |
| `lib_sample_len[index]` | Set to `0` |
| No error message logged | None |
| No exception thrown | None |
| Playback of failed sample | Track silently does nothing (checked at line 547-550) |

### Playback Safety Check

When a track attempts to play a sample:
```cpp
if (tr->sample_id>=0 && tr->sample_id<MAX_SAMPLES)
    tr->sample_end = lib_sample_len[tr->sample_id];
else
    tr->sample_id = -1;  // Mark as invalid
```

If a sample failed to decode (has ID but `lib_sample_data[id]` is null), the track is marked invalid and silently produces no audio.

### Assessment

The error handling is **minimal and silent**:
- No logging or user feedback on decode failure
- Corrupted/missing Ogg files simply result in no sound
- No fallback to a default sound
- Production builds would benefit from logging/assertions for debugging audio issues
