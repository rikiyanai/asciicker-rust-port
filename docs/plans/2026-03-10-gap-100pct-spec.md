## GAP 1 — C++ Engine Architecture (68% → 100%)

### 1.1 Sprite XP Blit (placeholder)

**Evidence:**
- `engine-port/src/render/sprite_blit.rs:99-128` — `blit_sprite()` doc says "Phase 5 placeholder sprite blit". Function body writes a single `b'S'` (0x53) with yellow foreground and dark-purple background at `(entry.screen_x, entry.screen_y)`.
- `engine-port/src/render/sprite_blit.rs:104-107` — `_sample_buffer` parameter is explicitly unused (prefixed `_`). Depth testing against the SampleBuffer is NOT performed.
- `engine-port/src/render/pipeline.rs:674-679` — Post-RESOLVE blit loop calls `blit_sprite` for every drained queue entry. The loop is correct; the called function is the stub.
- `engine-port/src/render/sprite_blit.rs:24-33` — `SpriteRenderEntry` has `sprite_name: String`, `anim: u32`, `frame: u32`. These fields are populated in `pipeline.rs:452-461` but are never read inside `blit_sprite`.
- `engine-port/src/asset_loader/xp_sprite.rs:109-179` — `parse_xp()` can fully parse .xp files into `XpSprite { layers }`.
- `engine-port/src/asset_loader/xp_sprite.rs:240-319` — `merge_layers()` implements all layer semantics: L0 colorkey, L1 height, L2 visual base, L3..N-2 intermediate overwrites, L(N-1) swoosh merge.

**Gap:** `blit_sprite` does not:
1. Load the `XpSprite` asset by `sprite_name`
2. Determine the animation/frame slice within the merged layer data
3. Project each cell of the sprite onto the screen with correct offsets
4. Respect depth: skip cells whose sample buffer height < the sprite's depth
5. Perform the transparent-cell skip (cells where `MergedCell.transparent == true`)

**Requirements for 100%:**

| REQ | Description | Acceptance |
|-----|-------------|------------|
| S-01 | `blit_sprite` receives or looks up `&XpSprite` via a `Res<Assets<XpSprite>>` pass-through from the pipeline | No more `sprite_name: String` lookup failure; assert `Option` unwrap does not panic for loaded sprites |
| S-02 | Function calls `merge_layers(&sprite)` to get the flat `Vec<MergedCell>` | Merged cell count == `sprite.width * sprite.height` |
| S-03 | Animation frame offset computed as `frame_idx * sprite.width * sprite.height` within merged cell array | Correct frame drawn for `anim=0,frame=0` vs `anim=0,frame=1` |
| S-04 | Each non-transparent cell is written to `AsciiCellGrid` at `(screen_x + dx, screen_y + dy)` with `merged_cell.glyph as u16`, `merged_cell.fg`, `merged_cell.bg` | Visual matches REXPaint preview of .xp file |
| S-05 | Cell is skipped when `MergedCell.transparent == true` | Background terrain/mesh visible through transparent sprite cells |
| S-06 | Depth test: cell is skipped when `sample_buffer.sample_at(sx, sy).height < entry.dist` (approximated world depth) | Sprites behind walls do not bleed through |
| S-07 | `_sample_buffer` parameter is now used (remove `_` prefix) | Compiler warning gone |
| S-08 | Bounds check covers full sprite rectangle `[screen_x, screen_x+w) x [screen_y, screen_y+h)`, not just origin | No panic on partial off-screen sprites |
| S-09 | Unit test: `blit_sprite` for a 1x1 sprite with known glyph+fg+bg writes correct cell | `grid.cell_at(sx, sy).glyph == expected_glyph` |
| S-10 | Unit test: transparent cell leaves prior grid cell unchanged | Prior cell not overwritten |

## GAP 2 — Alex Harri System (82% → 100%)

### 2.1 No GPU Sampling Path

**Evidence:**
- `engine-port/src/render/shape_vector.rs:1-11` — Module doc explicitly states: "The runtime path stays CPU-side and feeds the existing `GlyphSelector` extension point at resolve time."
- `engine-port/src/render/shape_vector.rs:487-568` — `ShapeVectorGlyphSelector::select_glyph()` runs entirely in the Bevy `PostUpdate` main-thread context. All bilinear sampling, circular region sampling, and k-d tree lookup happen on CPU.
- `engine-port/src/render/shape_vector.rs:419-447` — `sample_circular_region` iterates `quality` times (default 8) per point, per cell, calling `bilinear_sample_lightness` each time. This is `O(cells * 6 points * 8 samples)` per frame on CPU.
- `engine-port/src/render/pipeline.rs:597-634` — The resolve loop iterates every cell and calls `shape_sel.select_glyph(...)` synchronously in a tight loop.

**Gap:** No GPU compute path exists. At 240×135 = 32,400 cells × 6 points × 8 samples = ~1.5M bilinear lookups per frame on CPU. No WGSL compute shader for sampling; no async GPU readback path.

**Requirements for 100% (optional GPU path, not blocking):**

| REQ | Description | Acceptance |
|-----|-------------|------------|
| G-01 | Define a WGSL compute shader `shape_vector.wgsl` that takes the SampleBuffer (as R32Float texture) and outputs a glyph-index texture | Shader compiles without WGSL errors |
| G-02 | Compute shader implements the same 6-point circular multi-sampling as `sample_circular_region` | Same glyph output as CPU path for identical inputs (unit test with known buffer) |
| G-03 | Compute shader implements directional crunch and global crunch via push constants matching `ShapeVectorConfig` | Crunch output matches CPU reference to within floating-point tolerance |
| G-04 | GPU path is feature-gated (`features = ["shape_vector_gpu"]`); CPU path remains the default | Both paths compile; `cargo test` passes for both |
| G-05 | GPU path reads back glyph-index texture asynchronously (frame N-1 result used for frame N to avoid stall) | No pipeline stall visible in frame timing (`timing.resolve_us` drops by ≥ 50% on GPU path) |

**Note:** G-01 through G-05 are OPTIONAL. The CPU path is correct and sufficient for parity. GPU path is a performance enhancement only.

### 2.2 No Alphabet Switching

**Evidence:**
- `engine-port/src/render/shape_vector.rs:26` — `const DEFAULT_ALPHABET_JSON: &str = include_str!("alphabets/default.json");` — single alphabet embedded at compile time.
- `engine-port/src/render/shape_vector.rs:89-92` — `default_alphabet()` returns `&'static DefaultAlphabetData` from a `OnceLock`. Once initialized, it cannot be replaced.
- `engine-port/src/render/shape_vector.rs:304-313` — `ShapeVectorConfig` has no alphabet field; only crunch/threshold/quality parameters.
- `engine-port/src/render/shape_vector.rs:313-330` — `ShapeVectorMatcher::new_default()` always uses `default_alphabet().entries`. No variant for other alphabets.
- `engine-port/src/render/shape_vector.rs:318-331` — `ShapeVectorMatcher::new(characters: &[CharacterEntry])` is generic but never called with a non-default alphabet from the pipeline.
- `engine-port/src/render/shape_vector.rs:487-498` — `ShapeVectorGlyphSelector` hardcodes `default_alphabet()` inside `select_glyph` at lines 528, 541.

**Requirements for 100%:**

| REQ | Description | Acceptance |
|-----|-------------|------------|
| A-01 | `ShapeVectorConfig` gains an `alphabet: AlphabetId` field, where `AlphabetId` is an enum or string key | Field present and serializable |
| A-02 | A registry `AlphabetRegistry: Resource` maps `AlphabetId → DefaultAlphabetData` loaded from JSON files in `assets/alphabets/` | `AlphabetRegistry::get(id)` returns `Option<&DefaultAlphabetData>` |
| A-03 | `ShapeVectorGlyphSelector::select_glyph` reads alphabet from registry via `config.alphabet` instead of hardcoding `default_alphabet()` | Changing `config.alphabet` at runtime causes different glyphs to be selected |
| A-04 | `ShapeVectorMatcher` is rebuilt (or swapped) when `config.alphabet` changes; k-d tree reflects new alphabet entries | No stale-tree bug when alphabet is switched |
| A-05 | At least two alphabets loadable from `assets/alphabets/`: `default.json` and a minimal test alphabet | Both load without panic |
| A-06 | Unit test: switch alphabet mid-run; verify `select_glyph` returns glyphs from new alphabet's entry set | Test passes |

## GAP 3 — Mage Core Output Layer (90% → 100%)

**Evidence:**
- `engine-port/src/output/gpu_plugin.rs:309-320` — Inside `prepare_ascii_textures`, a new uniform buffer is created unconditionally every frame:
  ```rust
  let uniform_buffer = render_device.create_buffer_with_data(&BufferInitDescriptor { ... });
  textures.uniform_buffer = Some(uniform_buffer);
  ```
  This overwrites the previous buffer handle every frame even when font dimensions haven't changed.

**Requirements for 100%:**

| REQ | Description | Acceptance |
|-----|-------------|------------|
| M-01 | Uniform buffer is created once (on first prepare or on font dimension change) and reused via `render_queue.write_buffer()` for subsequent frames | `render_device.create_buffer_with_data` called ≤ once per font-config change, verified by counter in test |

### 3.2 glyph_index Truncated to u8

**Evidence:**
- `engine-port/src/output/gpu_types.rs:64-69` — `char_data` encodes glyph indices as `idx as u8` with `debug_assert!(idx <= 255)`. The `AsciiCellGrid.char_indices` field is `Vec<u16>` but only the lower 8 bits reach the GPU.
- `engine-port/src/output/ascii_cell_grid.rs` (not read but implied) — `char_indices: Vec<u16>`.

**Gap:** CP437 only needs 0-255 so u8 is correct for the current alphabet. However `char_indices` being `u16` with silent truncation to `u8` is a silent data contract violation that will cause invisible bugs if any glyph > 255 is ever stored.

**Requirements for 100%:**

| REQ | Description | Acceptance |
|-----|-------------|------------|
| M-02 | `AsciiCellGrid.char_indices` is changed from `Vec<u16>` to `Vec<u8>`, OR a non-debug assertion replaces the `debug_assert!` with a hard error | No silent truncation path exists |

## GAP 4 — ECS Migration (72% → 100%)

### 4.1 RuntimeTerrain — Monolithic Singleton

**Evidence:**
- `engine-port/src/terrain/mod.rs:29-43` — `RuntimeTerrain: Resource` is a single Bevy resource containing:
  - `root: Option<QuadNode>` (the entire quadtree in one allocation)
  - `patch_map: HashMap<(i32, i32), RuntimePatch>` (all patch data flat)
  - `patch_count: usize`
- There are no `TerrainPatch` components; patches are not ECS entities.
- The quadtree traversal is a recursive function on a plain Rust `QuadNode` enum, not a Bevy query.

**Gap:** `RuntimeTerrain` is a C-style data container wrapped in a Bevy resource. The ECS cannot query individual patches, attach per-patch components (LOD state, load state, visibility), or parallelize patch processing with `par_iter`.

**Requirements for 100%:**

| REQ | Description | Acceptance |
|-----|-------------|------------|
| E-01 | Each terrain patch is spawned as a Bevy entity with at minimum: `TerrainPatchComponent { x: i32, y: i32, data: Arc<RuntimePatch> }` | `world.query::<&TerrainPatchComponent>().iter().count() == terrain.patch_count` |
| E-02 | Frustum culling is implemented as a Bevy system using `Query<(&TerrainPatchComponent, &Aabb)>` rather than the monolithic `RuntimeTerrain::query_visible` | Existing tests still pass with entity-based path |
| E-03 | `RuntimeTerrain` is removed or reduced to a lightweight index (root bounds + level) that does NOT own patch data | `RuntimeTerrain` struct has no `patch_map` or `QuadNode` tree |
| E-04 | LOD state can be attached per-patch as a separate component without modifying `TerrainPatchComponent` | `commands.entity(patch_entity).insert(LodState::High)` compiles and is queryable |

### 4.2 RuntimeWorld — Monolithic Singleton

**Evidence:**
- `engine-port/src/world/mod.rs:31-38` — `RuntimeWorld: Resource` holds:
  - `bsp_root: Option<BspNode>` (entire BSP tree)
  - `instances: Vec<RuntimeInstance>` (all instances flat by index)
  - `flat_list: Vec<InstanceId>` (non-BSP instances)
- `engine-port/src/world/mod.rs:34-35` — `instances: Vec<RuntimeInstance>` — `RuntimeInstance` is an enum variant (Mesh / Sprite / Item) with all data inlined.
- `engine-port/src/render/pipeline.rs:414-486` — Stage 3 WORLD iterates `visible_instances` from `world_data.query_visible(...)` and dispatches on `VisibleInstance::Mesh` vs `VisibleInstance::Sprite`. This is a loop over raw IDs, not a Bevy query.

**Gap:** World instances are not ECS entities. Per-instance state (animation frame, position, yaw) cannot be updated via Bevy queries or change detection. Systems cannot filter by component.

**Requirements for 100%:**

| REQ | Description | Acceptance |
|-----|-------------|------------|
| E-05 | Each world instance is spawned as a Bevy entity with a `WorldInstanceComponent` marker and type-specific components: `MeshInstance { mesh_id, tm }`, `SpriteInstance { sprite_name, pos, yaw, anim, frame }`, or `ItemInstance { pos, yaw }` | `world.query::<&SpriteInstance>().iter().count() > 0` after world load |
| E-06 | The BSP query is replaced or augmented by a Bevy-compatible visibility system that sets an `InView` marker component on visible entities | Pipeline Stage 3 iterates `Query<&MeshInstance, With<InView>>` instead of raw VisibleInstance |
| E-07 | `RuntimeWorld` is removed or reduced to a lightweight BSP index resource (no `instances` Vec) | `RuntimeWorld.instances` field removed |
| E-08 | Animation frame update system uses `Query<&mut SpriteInstance, With<Character>>` rather than operating on `PhysicsIO` → `SpriteRenderEntry` indirection | Change detection on `SpriteInstance.frame` works correctly |

### 4.3 SampleBuffer — Monolithic Singleton

**Evidence:**
- `engine-port/src/render/pipeline.rs:279-294` — `render_pipeline_system` takes `mut sample_buffer: ResMut<SampleBuffer>` — the entire CPU raster buffer is one globally mutable resource.
- `engine-port/src/render/pipeline.rs:386-502` — The six render stages share the same `SampleBuffer` through nested scopes; the borrow is explicitly managed with comments ("mutable borrow of sample_buffer.samples ENDS here").
- No parallelism across the buffer is possible within the pipeline because of the single `ResMut<SampleBuffer>` borrow.

**Gap:** `SampleBuffer` is a single flat allocation. Potential tile-parallel rasterization is blocked because Rust's ownership model prevents concurrent `&mut` slices without unsafe code or explicit splitting.

**Requirements for 100%:**

| REQ | Description | Acceptance |
|-----|-------------|------------|
| E-09 | `SampleBuffer` exposes a `split_tiles(n: usize) -> Vec<SampleTile>` method that produces non-overlapping `&mut [Sample]` slices | Method compiles; slices cover the full buffer without overlap |
| E-10 | Terrain stage (Stage 2) uses `par_iter_mut` or `rayon` to rasterize patches in parallel across tiles | `timing.terrain_us` decreases on multi-core machines |
| E-11 | Tile split is consistent with the resolve stage: each tile's samples are independently resolvable to a sub-grid of `AnsiCellGrid` | Resolve output identical to single-threaded path (deterministic test) |

### 4.4 PhysicsIO — Shared Mutable State

**Evidence:**
- `engine-port/src/game/mod.rs:20` — `use crate::physics::{PhysicsIO, PhysicsState};`
- `engine-port/src/character/mod.rs:21` — `use crate::physics::PhysicsIO;`
- Systems in `game/mod.rs` use `ResMut<PhysicsIO>` for torque, velocity, and position data shared between character logic and camera sync.

**Gap:** `PhysicsIO` acts as a shared mutable message bus between systems (character input → physics → camera). This is a C-style callback pattern, not ECS data-flow. It creates implicit ordering constraints and makes it impossible to have multiple physics entities.

**Requirements for 100%:**

| REQ | Description | Acceptance |
|-----|-------------|------------|
| E-12 | Per-entity physics state is represented as ECS components: `PhysicsBody { vel: Vec3, grounded: bool }` and `PhysicsForces { impulse: Vec3, torque: f32 }` | `Query<(&PhysicsBody, &PhysicsForces), With<Character>>` returns player entity |
| E-13 | `PhysicsIO` resource is removed; camera sync system reads from `Query<(&PhysicsBody, &Transform), With<PlayerMarker>>` | `PhysicsIO` struct deleted from codebase |
| E-14 | Multiple characters or NPCs can each have independent `PhysicsBody` without sharing a singleton | Spawn two `Character` entities; each has independent physics state |

## Acceptance Gate for "100% Aligned"

All four axes reach 100% when:

1. `cargo test --workspace` passes with zero failures.
2. A test scene runs for 60 frames; glyph occupancy (non-space cells / total cells) ≥ 40% (threshold to be calibrated against C++ reference output).
3. A visible sprite (player character) renders with correct XP frame, not 'S'.
4. Water surface reflects both terrain AND meshes.
5. Deterministic trace comparison against C++ reference output within ≤ 5% cell discrepancy (pending T-01 C++ build unblock).
6. Alphabet switching works at runtime without crash.
7. All ECS requirements (E-01..E-14) pass their stated acceptance tests.
